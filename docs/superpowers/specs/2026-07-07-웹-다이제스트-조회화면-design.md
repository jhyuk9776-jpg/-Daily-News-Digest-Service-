# 웹 다이제스트 JSON 계약 + 최소 조회화면 (설계)

- 작성일: 2026-07-07 / 이재혁 · Claude
- 로드맵 위치: 07-07 "Next.js(PWA) 뼈대 + 다이제스트 조회 화면(정적 JSON)"
- 관련: [일정-로드맵 §4a·5](../../../기획/일정/일정-로드맵.md), [결정 로그](../../../기획/decision-log.md)

## 배경 / 문제

파이프라인의 최종 출력은 `News/YYYY-MM-DD.md`(사람용 마크다운) 하나뿐이다.
웹(프로그램)이 이 마크다운을 파싱해 의미를 되짚는 것은 형식 변경에 취약하다.
로드맵 M1(07-06)의 "웹 JSON 계약 확정"이 실제로는 미완이라, 웹 조회화면 착수의
선결 과제로 남았다. 이 설계는 **파이프라인이 구조화 JSON을 내보내는 계약을 확정**하고,
그 계약을 읽는 **최소 Next.js 조회화면**을 만든다.

## 범위

### 하는 것 (오늘, 07-07)
1. 다이제스트 JSON 계약(스키마) 확정
2. 파이프라인이 `web/public/data/<date>.json` + `latest.json` 출력 (마크다운은 그대로 유지)
3. 그 JSON을 읽어 분야별로 렌더하는 최소 Next.js 조회화면

### 안 하는 것 (YAGNI — 각각 로드맵 뒷날)
- 날짜 네비게이션 / 과거 보기 / `index.json`
- PWA(매니페스트·서비스워커·오프라인·설치)
- 경제 표시개수 조절 UI · 관심분야 필터
- Vercel 실제 배포 · 도메인 연결
- 객관성 점수 노출 · 디자인 폴리시

## 아키텍처 / 데이터 흐름

```
파이썬 파이프라인 (src/summarize.py)
   ├─ News/2026-07-07.md              (기존, 사람용 — 유지)
   └─ web/public/data/2026-07-07.json (신규, 웹용 계약)
        + web/public/data/latest.json  (항상 최신 = 오늘자 동일 내용)
                    │
              cron 커밋 → Vercel 재빌드 (배포는 이번 범위 밖)
                    │
   Next.js 웹 (web/) — latest.json 읽어 분야별 조회화면 렌더
```

"오늘자 1개"를 웹이 어떻게 아는가 → 파이프라인이 날짜파일과 함께 `latest.json`(최신 동일본)도
쓴다. 웹은 `latest.json`만 읽으므로 날짜 하드코딩·index가 불필요하다. 날짜파일이 날짜별로
쌓여 있으므로, 나중에 날짜 네비게이션을 붙일 때 계약을 바꾸지 않고 `index.json`만 추가하면 된다.

## JSON 계약 (스키마)

```json
{
  "date": "2026-07-07",
  "categories": [
    {
      "name": "경제",
      "items": [
        {
          "title": "네이버파이낸셜·두나무 합병 또 연기",
          "bullets": ["네이버파이낸셜과 두나무의 합병이 연기되었다."],
          "source": "한국경제",
          "link": "https://www.hankyung.com/article/2026070686961",
          "related_links": [{"source": "매일경제", "link": "https://..."}]
        }
      ]
    }
  ]
}
```

| 필드 | 타입 | 의미 |
|---|---|---|
| `date` | string (YYYY-MM-DD) | 다이제스트 날짜 |
| `categories[]` | array | 분야 목록. 순서는 마크다운과 동일(경제→사회→세계→IT/테크) |
| `categories[].name` | string | 분야명 |
| `categories[].items[]` | array | 해당 분야 기사 항목 |
| `items[].title` | string | 기사 제목 |
| `items[].bullets[]` | string[] | 사실 중심 요약 불릿. 없으면 빈 배열 |
| `items[].source` | string | 대표 매체명 |
| `items[].link` | string (URL) | 대표 원문 링크 |
| `items[].related_links[]` | array | 같은 사건 관련보도 `{source, link}`. 없으면 빈 배열 |

- 화면의 "외 관련 N건"은 웹이 `related_links.length`로 계산(계약에 count 중복 저장 안 함).
- 불릿 0개 항목의 취급은 **마크다운 렌더와 동일 규칙**을 따른다(같은 소스에서 파생하므로 자동 일치).

## 파이프라인 변경 (작게, 격리)

현재 `build_markdown(selected, results, counters)`이 `selected`+`results`를 훑어 마크다운을
조립한다. 이 **훑는 로직을 `collect_digest(selected, results) → 구조화 dict`** 로 추출한다.

- 마크다운 렌더러와 JSON 직렬화가 **같은 구조를 공유** → 둘이 어긋날 수 없다(단일 소스 파생).
- `collect_digest`가 낸 구조를 (a) 기존 마크다운 렌더가 소비, (b) `json.dump`로
  `web/public/data/<date>.json`과 `latest.json`에 저장.
- 마크다운 출력 경로·형식은 불변. 이메일 발송(notify) 경로도 불변.

### 단위 경계
- `collect_digest`: selected+results → 계약 구조 dict. 순수 함수(파일·네트워크 없음), 테스트 쉬움.
- JSON 쓰기: `collect_digest` 결과 + 출력 경로 → 파일 2개. 얇은 IO 래퍼.

## 웹 구조 (web/, Next.js App Router + TypeScript)

```
web/
  app/page.tsx           최신 다이제스트 페이지(서버 컴포넌트, 빌드 시 latest.json 읽음)
  components/Digest.tsx  다이제스트 하나(date-주입형)를 분야별로 렌더 — 재사용 단위
  lib/types.ts           계약과 1:1 대응하는 TypeScript 타입
  public/data/           파이프라인이 JSON 떨구는 자리 (.gitkeep + 실제 날짜/latest json)
```

- `app/page.tsx`는 빌드 시 `public/data/latest.json`을 파일시스템에서 읽어(SSG) `Digest`에 넘긴다.
  cron이 새 JSON을 커밋하면 Vercel 재빌드가 최신 데이터를 반영한다.
- `Digest`는 "다이제스트 데이터 하나를 받아 그린다"로 짜서, 나중 날짜 네비게이션이 위에
  얹혀도 그대로 재사용된다. 분야 헤더 → 기사 제목 → 불릿 → 출처 링크(+관련 N건).
- 스타일링은 최소·가독 수준만. 디자인 폴리시는 로드맵 M2 몫.

## 테스트

- **파이썬:** `collect_digest`가 기대 구조를 내는지(분야 순서·필드·빈 배열 규칙) + JSON 파일이
  스키마 키대로 써지는지. 기존 `tests/` unittest에 추가.
- **웹:** `Digest` 컴포넌트가 제목·불릿·출처를 렌더하는지 렌더 테스트 1개 + `next build` 통과.
  무거운 E2E는 로드맵 M4 몫.

## 리스크 / 열린 점

- Next.js/PWA 첫 셋업 트러블(로드맵 §5에서 예고된 밀림 지점) → 오늘 안 끝나면 07-08로 흡수.
- `latest.json` 중복 저장은 의도적(날짜파일과 동일본). 용량 미미, 웹 단순화가 이득.
- 웹 테스트 러너(예: vitest/jest) 도입은 최소 구성으로. 과설정 지양.
