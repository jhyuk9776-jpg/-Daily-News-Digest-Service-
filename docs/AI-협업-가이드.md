# AI 협업 공용 가이드

AI(코딩 에이전트·LLM)와 **어떤 작업이든** 함께할 때 꺼내 쓰는 공용 플레이북.
특정 프로젝트에 매이지 않는다. 여기 담는 것은 세 가지다.

1. **원칙** — AI를 효율적으로 부리는 핵심 사고방식
2. **막는 실패 · 원리** — 각 기법이 *어떤 실패를 막는지*와 *왜 통하는지*
3. **선례 로그** — 실제로 잘 풀린 사례를 append-only로 축적한다. 좋은 아이디어·효율적 해결이 나오면 계속 추가해 함께 성장시킨다.

> 사용법: 새 작업을 AI에게 맡기기 전에 §1~§3을 훑고, 프롬프트를 §2 체크리스트로 점검한다.
> 작업이 끝나면 배운 것을 §4 선례 로그에 한 항목 추가한다.

---

## 1. 핵심 원칙 3가지

이것만 기억하면 절반은 된다.

1. **추측을 제거하라.** AI의 실수 대부분은 "몰라서 상상한 것"에서 나온다.
   파일·함수·실제 데이터·용어를 구체적으로 주면 AI는 상상 대신 사실로 일한다.
2. **정답이 아니라 채점표를 줘라.** "무엇을 하라"보다 "무엇이 충족되면 성공인지 + 어떻게 검증하는지"를 주면
   AI가 스스로 결과를 검증하고, 증거 없이 "완료"를 선언하지 못한다.
3. **되물을 지점을 지정하라.** 판단이 갈리는 곳(제품·정책·되돌리기 어려운 결정)은 AI가 멋대로 정하지 않게
   게이트를 걸어라. 방향이 틀어졌을 때 코드를 갈아엎기 전에 잡힌다.

---

## 2. 프롬프트 6-블록 체크리스트

좋은 코딩/작업 프롬프트는 아래 6블록을 갖춘다. 각 블록은 **특정 실패**를 막는다.

| 블록 | 내용 | 막는 실패 | 원리 |
|---|---|---|---|
| **① 역할 + 맥락** | 무슨 시스템인지, 봐야 할 파일·함수명(`file.py:func`) | AI가 코드베이스를 헤매며 엉뚱한 곳을 뒤짐 | 구체적 앵커를 주면 탐색 비용이 사라진다 |
| **② 실제 예시/데이터** | 진짜 입력·출력·재현 사례 | 추상적 지시 → 문제를 오해 | LLM은 구체 예시로 문제의 경계를 잡는다. "고쳐줘"보다 "이 3개가 왜 이런지"가 훨씬 정확 |
| **③ 제약(하지 말 것)** | 금지 사항·트레이드오프 방향(예: 재현율보다 정밀도, 유료 API 금지) | 과잉 엔지니어링·원치 않는 의존성·엉뚱한 최적화 | 안 되는 걸 명시하면 탐색 공간이 좁아져 답이 빨라진다 |
| **④ 성공 기준 + 검증법** | 무엇이 충족되면 끝인지 + 어떻게 확인하는지(before/after 수치 등) | "됐습니다"인데 실제론 안 됨 | 스스로 채점하게 만든다. 증거를 요구하면 허위 완료가 불가능해진다 |
| **⑤ 되물을 지점** | 사람 판단이 필요한 결정을 명시 | AI가 임의로 정책/제품 결정을 내려버림 | 판단 분기점을 사람에게 넘기게 못박는다 |
| **⑥ 작업 방식(게이트)** | "분석 → 접근법 비교 → 승인 → 구현" 같은 단계 | 승인 없이 바로 코드 갈아엎음 | 단계 게이트가 조기 이탈을 잡는다 |

### 미니 템플릿

```
[역할/맥락] 너는 __ 을 다루는 __ 다. 관련 코드: file.py:func — 지금 이렇게 동작한다.
[문제/예시] 실제 사례: "____". 기대: ____ / 실제: ____.
[요구사항] 1) __ 2) __ (트레이드오프 방향: __ 우선)
[제약] __ 없이, __ 를 따르고, 변경은 최소 침습적으로.
[성공 기준] __ 이면 성공. before/after __ 를 보여줘라.
[되물을 것] __ 는 제품 결정이니 옵션을 제시하고 내 승인을 받아라.
[작업 방식] 코드 수정 전에 (1) 원인분석 (2) 접근법 2~3개 비교 (3) 추천 — 을 먼저 제시하고 확인받아라.
```

## 3. AI와 일할 때 고려할 것 (원칙 밖의 실전 감각)

- **작은 단위로 쪼개 맡겨라.** 한 번에 하나의 명확한 목적. AI는 한눈에 들어오는 범위에서 추론과 편집이 정확하다.
  파일/작업이 커지면 "너무 많은 걸 하고 있다"는 신호다.
- **컨텍스트를 최신·정확하게.** 오래된 파일 내용이나 기억에 의존하지 말고, 지금 상태를 읽혀라(코드는 진화한다).
- **불확실은 드러내게 하라.** "가정을 먼저 나열하라"고 시키면 잘못된 전제를 조기에 잡는다.
- **되돌리기 어려운 행동은 확인부터.** 삭제·덮어쓰기·외부 전송·배포는 승인 게이트를 건다.
- **실패를 버리지 마라.** 실패·오탐·헛수고도 기록하면 자산이 된다(무엇이·왜 실패했는지). §4에 남긴다.
- **검증을 코드가 아니라 결과로 요구하라.** "테스트 통과"라는 말이 아니라 실제 출력·수치·before/after로 확인한다.

### 2026-07-10 — "프롬프트를 먼저 설계"가 구현보다 빨랐다 (이재혁)
- **상황:** 다이제스트 중복 제거 문제를 AI에게 맡기려 했다.
- **효과적이었던 것:** 곧장 코드를 고치는 대신, 실제 다이제스트에서 **재현 사례(광주 토허구역 3건 중복)**를
  먼저 뽑고, 관련 파일·함수(`curate.py:cluster_articles`)와 성공 기준(before/after 클러스터 수)을 담은
  프롬프트를 설계했다. 문제의 경계가 선명해져 접근법 비교가 곧바로 가능해졌다.
- **배운 원리:** 원칙 ①(추측 제거)·②(채점표). 재현 사례 하나가 추상적 지시 열 줄보다 강하다.

### 2026-07-10 — 실데이터 회귀 점검이 "승인된 접근법"의 함정을 잡았다 (이재혁)
- **상황:** 중복 제거를 어휘(토큰 겹침) 방식으로 구현하기로 승인하고, TDD로 만들어 단위테스트 6/6·전체
  120/120 통과까지 갔다. 여기서 멈췄다면 "완료"로 커밋했을 것이다.
- **효과적이었던 것:** 커밋 전에 **실제 코퍼스(07-03 경제 369건)로 before/after를 돌려봤다.** 그랬더니
  서로 다른 사건 56~99건이 한 덩어리로 뭉치는 치명적 과잉병합이 드러났다. 단위테스트는 초록불인데
  실데이터는 붕괴 — 테스트 통과 ≠ 올바른 동작. 곧바로 df·희귀어 등 변형을 실측 비교해 "어휘/빈도로는
  구조적으로 불가"임을 수치로 확정하고, 방향을 임베딩(Phase 2)으로 틀었다.
- **배운 원리:** 원칙 ②(정답이 아니라 채점표). "테스트 통과"가 아니라 **실제 출력·수치**로 검증하라(§3 마지막 줄).
  단위테스트는 내가 상상한 케이스만 지킨다. 실데이터가 진짜 채점관이다. 승인받은 접근법이라도 데이터가
  반증하면 멈추고 되돌아가는 것이 정답보다 빠르다.
