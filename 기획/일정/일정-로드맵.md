## 7. 전달 채널(이메일→카톡) — 아이디어 기록 (결정 대기, 2026-07-03)

**요구:** 내일부터 매일 아침 **Gmail 수신**. 읽기 쉬운 형태로, **PC·모바일 카카오톡 양쪽에서 바로 복사-붙여넣기** 가능.

**핵심 긴장:** 카카오톡은 **서식 없는 plain text**라 마크다운/HTML 서식이 복붙 시 깨진다.
→ 본문은 **유니코드 기호(▪ 불릿, ─ 구분선)를 쓴 plain text**가 카톡 복붙에 최적. 이메일은 HTML로 예쁘게 하되
   카톡용 plain 블록을 **병행**(멀티파트)하면 PC 가독성과 카톡 호환을 둘 다 챙길 수 있다.

**발송 방법 후보:**
| 안 | 방법 | 장점 | 단점 |
|---|---|---|---|
| A | GitHub Actions에서 SMTP 발송(Gmail 앱비밀번호 or `dawidd6/action-send-mail`) | cron 끝에 붙이면 끝, 서버 불필요 | 앱비밀번호를 Secret로 관리 |
| B | 파이프라인 마지막 단계에서 `smtplib` 발송 | 로컬/CI 공용, 코드 일원화 | 자격증명 관리 동일 |
| C | 카톡 직접 발송(나에게 보내기 API) | 복붙 불필요 | 비즈니스 채널·토큰 등 제약 커 **후순위** |

**포맷 후보:** (a) plain text 단일(카톡 최적) / (b) HTML+plain 멀티파트(PC 가독성+카톡 호환). 출처 URL은 그대로 노출.

→ **추천(잠정):** 발송 **A안(Actions SMTP)** + 포맷 **(b) 멀티파트**, 카톡용 plain 블록을 메일 하단에 별도 제공.
   **결정은 07-06에.** (이 절은 기록용 — 확정 아님.)

## 9. Phase 2 상세: 매체 객관성 동적 점수 (로드맵 원안)

로드맵 4절 "출처 신뢰도 티어"를 **동적 객관성 점수**로 확장한 아이디어(2026-07-01).
Phase 1의 "객관적 서술 우선" 원칙([결정 로그 1.3](../decision-log.md))에 숫자 기반 학습을 붙여,
매체 신뢰도가 피드백으로 수렴하게 한다. **구현 상세·현재 진행분은
[시스템기획/기능설계/04-객관성-점수축적기](../시스템기획/기능설계/04-객관성-점수축적기.md)** 참조
(이 절은 방향·원안 기록).

### 초안 아이디어(사용자 제안)

- 모든 매체 100점에서 시작한다.
- 객관적이라 피드백받은 매체는 가점(예: +1), 감정적/평가 표현이 나오면 감점.
- 일정 점수(예: 80점) 이하로 내려가면 제외 후보로 두고 새 매체를 찾는다.
- 좋게 평가된 매체는 점수가 올라 신뢰도가 누적된다.
- 감점 트리거 예: "~가 다 했네"처럼 평가어가 제목/본문에 포함되면 감점
  (감점 사전은 AI_CONTEXT §6 "피해야 할 표현"에서 시드).

### 구현 전 정할 점(다듬은 방향)

| 쟁점 | 문제 | 방향 |
|---|---|---|
| 점수 단위 | 좋은 매체도 특정 기사만 감정적일 수 있어, 기사 하나로 매체 전체를 깎으면 거칠다 | 기사별 객관성 점수로 선별하고, 그 결과를 매체 점수에 천천히 반영(느린 평판) |
| 누적 vs 비율 | 기사를 많이 내는 매체는 양 때문에 감점이 더 쌓임 | 누적합 대신 기사당 평균/이동평균으로 정규화 |
| 임계값·증감폭 | 80점·±1은 임의값. 회복·하락 속도가 결과를 좌우 | 초기엔 제외 없이 기록만 하다가 점수 분포를 보고 임계값 결정 |
| 새 매체 자동 탐색 | RSS 자동 발굴은 별개의 큰 기능 | 당분간 수동 교체(제외 후보만 사용자에게 알림) |
| 키워드 감점 오탐 | "충격" 등은 사실 기사(지진 충격 등)에도 등장 | 단순 매칭은 거칠므로 초기엔 보조 신호로만 사용 |

### 연료(선행 작업)

이 시스템은 라벨 데이터가 연료다. Phase 1 동안 "객관적 예시 수집"(결정 로그 1.3의
네이버 링크 1호처럼 사용자가 좋다고 고른 기사)을 꾸준히 모아두면, Phase 2에서 임계값과
감점 사전을 실제 데이터로 튜닝할 수 있다. 실제 구현 시 별도 브레인스토밍을 거친다.

### 빠른 안정화: 전량 record-only 스코어링 (2026-07-01 방향, Phase 2 첫 착수 — 구현됨)

시간 제약상 객관성 점수를 빨리 수렴시키기 위해, 요약 대상 8건뿐 아니라 **수집·중복제거된
전량**(하루 ~1000건)의 제목·RSS 요약에 객관성 휴리스틱(AI_CONTEXT §6 평가어 감점 +
숫자·인용 가점)을 적용해 **점수만 기록**한다. 요약(AI 호출) 없이 확인만 하므로 비용 0.

| 항목 | 방향 |
|---|---|
| 저장 | 기사별 점수 + 매체별 이동평균을 **로컬 JSON**에 누적. Supabase는 다중 사용자 시점으로 미룸 |
| 모드 | **observe-only** — 랭킹·제외에 반영하지 않고 분포를 볼 때까지 기록만 |
| 백필 | 기존 `raw/YYYY-MM-DD.json`에 소급 적용해 즉시 히스토리 확보 |
| 데이터 분리 | (a) 휴리스틱 자동점수=양(노이즈 있음), (b) 사용자 라벨=질(보정용) — 섞지 않고 따로 저장 |
| 한계 | RSS는 제목·리드 중심이라 본문 평가어는 놓침(부분 신호). 휴리스틱 오탐은 보조 신호로만, 자동 제외 금지 |
