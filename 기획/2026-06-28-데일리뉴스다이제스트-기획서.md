작성일: 2026-06-28

방법론: StoryBrand 7단계(SB7) 서사 + 기술 설계
참고: 『무기가 되는 스토리(Building a StoryBrand)』

## 0. 시작하기 전에 — 이야기의 3가지 핵심 질문

> 모든 좋은 이야기는 이 3가지에 답한다. 제품도 하나의 이야기다.

**i) 주인공이 원하는 게 무엇인가?**
주인공(= 재혁 / 정보를 얻으려는 사람)은 **매일 아침, 짧은 시간에 세상에서 실제로 일어난 일을 "판단·해석 없는 사실"로 빠르게 파악**하고 싶다.
더 깊은 욕구: 그렇게 잡은 세상의 흐름 위에서 **남보다 먼저 문제와 기회를 보는 사람**이 되는 것. (이번 Phase 1은 그 토대인 "사실 파악"까지를 책임진다.)

**ii) 주인공이 원하는 것을 얻지 못하도록 반대하는 세력은 누구인가?**
사람이 아니라 **3개의 힘**이다.
- **정보 과잉** — 뉴스는 하루에도 수백 건. 다 볼 수 없다.
- **편향·잡음** — 의견, 자극적 제목, 낚시, 매체별 프레임이 "사실"에 들러붙어 순수한 사건을 가린다.
- **시간 부족** — 매일 직접 여러 매체를 뒤져 교차검증할 시간이 없다.
> 한마디로 적은 **"편향된 정보 홍수"**다.

**iii) 원하는 것을 얻으면(혹은 못 얻으면) 주인공의 삶은 어떻게 달라지는가?**
- **얻으면:** 매일 5분으로 세상의 사실 골격을 잡는다. 정보에 휩쓸리는 대신, 흐름 *위에 서서* 문제·아이디어를 떠올리는 사람이 된다. 이 습관이 곧 기획자/창업가의 입력 루틴이 된다.
- **못 얻으면:** 알고리즘이 떠먹여 주는 편향된 피드만 보거나, 아예 흐름과 단절된다. 판단과 아이디어의 토대가 약해지고, "세상이 어디로 가는지 모르는" 상태가 누적된다.

## 1. SB7 — 일곱 단계 서사

### ① 캐릭터 (주인공)
**재혁** — 취업을 준비하며 기획/문제발굴 역량을 키우는 사람. 세상의 흐름을 읽고 그 위에서 사업·서비스 아이디어를 발견하고 싶어한다. 단, 매일 뉴스를 일일이 챙길 시간과 인내심은 없다.
> 주인공은 "뉴스를 많이 읽고 싶은 사람"이 아니라 **"세상을 효율적으로 파악해 그 위에서 생각하고 싶은 사람"**이다. 제품의 주인공은 정보가 아니라 사람이다.

### ② 난관에 직면한다
주인공은 매일 **"편향된 정보 홍수"**라는 적과 마주친다. 이 난관은 세 층위로 작동한다.
- **외적 문제:** 뉴스가 너무 많고 흩어져 있어 오늘 무슨 일이 일어났는지 한눈에 못 본다.
- **내적 문제:** "나만 흐름에서 뒤처지는 것 아닌가", "휩쓸리고 있다"는 불안과 피로.
- **철학적 문제:** 사람은 *누군가의 판단이 덧칠되지 않은, 일어난 그대로의 사실*을 볼 권리가 있다. 사실과 의견이 뒤섞인 정보 환경은 그 권리를 침해한다.

### ③ 가이드를 만난다
주인공을 구하는 영웅이 아니라 **가이드**로서, 이 시스템(데일리 사실 뉴스 다이제스트)이 등장한다. 가이드는 두 가지를 갖춰야 한다.
- **공감:** "매일 뉴스 다 볼 시간 없죠. 의견 말고 사실만 빠르게 보고 싶죠." — 주인공의 피로를 안다.
- **권위(신뢰):** 아무 소스나 긁지 않는다. **미리 합의해 고정한 신뢰 소스 목록**과 **'판단 없이 사실만'이라는 명문화된 요약 규칙**으로 일관된 품질을 보증한다.

### ④ 계획을 제시한다
가이드는 주인공에게 **단순하고 명확한 계획**을 준다. (복잡하면 주인공은 움직이지 않는다.)
**4단계 파이프라인:** `① 수집 → ② 선별 → ③ 요약 → ④ 전달`
주인공이 할 일은 단 하나 — **아침에 도착한 다이제스트를 5분간 읽는 것**. 나머지는 시스템이 한다.

### ⑤ 행동을 촉구한다
- **직접적 촉구:** "내일 아침부터 받아보기 시작하자." (자동 실행을 켠다)
- **전환적 촉구:** 이미 **C(맛보기)로 결과물의 맛을 봤다.** 다음 행동은 **A(로컬 스크립트)로 제대로 짓는 것.**

### ⑥ 실패를 피하게 도와준다 (걸려 있는 위험)
이 시스템이 없을 때 주인공이 겪는 실패:
- 정보 홍수에 빠져 **오늘 세상에 무슨 일이 있었는지조차 모르고** 하루를 보낸다.
- 알고리즘 피드의 **편향과 자극만 흡수**해 세상을 왜곡되게 인식한다.
- "언젠가 챙겨봐야지" 하며 **흐름에서 점점 멀어진다.**
시스템은 매일 아침 사실 골격을 떠먹여 줌으로써 이 실패들을 막는다.

### ⑦ 성공으로 끝맺는다
성공한 주인공의 모습:
- 매일 아침 **5분**, 분야별 사실 다이제스트로 세상의 골격을 잡는다.
- 정보에 휩쓸리지 않고 **흐름 위에서 문제·기회를 보는 입력 습관**을 갖는다.
- 이 토대 위에서 **Phase 2(불편/만족 신호 수집)**로 확장해, 사실 파악을 넘어 *아이디어 발굴*까지 나아간다.
> 성공의 그림은 "뉴스를 다 읽는 사람"이 아니라 **"적게 읽고 더 잘 아는 사람"**이다.

## 2. 기술 설계

### 2.1 범위 (Scope) / 비범위 (Non-goals)
- **이번에 만든다 (Phase 1):** 분야별 **사실 중심** 데일리 다이제스트. 판단·논평·개인화 없음. 매일 아침 자동 생성.
- **이번엔 안 만든다:** 불편/만족 신호 추출(별도 Phase 2), 의견·분석, 개인화 추천, 국내 커뮤니티 스크래핑.
- **설계 원칙:** 4단계를 **떼었다 붙였다 할 수 있는 부품**으로. 나중에 ③ 요약만 갈아끼우면 Phase 2가 되고, ① 수집에 소스만 추가하면 확장된다.

### 2.2 접근 방식 (확정: A)
**A — 로컬 파이썬 스크립트 + 스케줄러.** `fetch → summarize(LLM) → write` 를 하는 파이썬 프로그램을 만들고, 매일 아침 자동 실행한다.
선정 이유: 통제력·일관성 최고, AI 코딩 학습 + 포트폴리오 가치 최고, 비용은 요약 API 호출 소액뿐. (C는 맛보기로 이미 사용 완료.)

### 2.3 파이프라인 (4단계 부품)

| 단계 | 하는 일 | 입력 → 출력 | 부품 경계 |
|---|---|---|---|
| ① 수집 (fetch) | 고정 소스에서 어제~오늘 기사 수집 | 소스 목록 → 기사 리스트(제목·요지·시각·링크) | 소스 추가/교체가 여기서만 일어남 |
| ② 선별 (select) | 분야별로 오늘의 N개 선별·중복 병합 | 기사 리스트 → 추린 기사 리스트 | "무슨 기준으로 고르나"가 여기 한 곳에 |
| ③ 요약 (summarize) | LLM이 '판단 없이 사실만' 규칙으로 압축 | 추린 기사 → 분야별 사실 요약 | **Phase 2에서 이 부품만 교체** |
| ④ 전달 (deliver) | 날짜별 마크다운으로 저장(+선택: 알림) | 요약 → `News/YYYY-MM-DD.md` | 전달 채널 추가가 여기서만 |

> 각 부품은 독립적으로 테스트 가능해야 한다 — ①만 돌려 "기사를 잘 가져오나", ③만 돌려 "요약 규칙을 잘 지키나" 확인할 수 있게 분리한다.

### 2.4 '판단 없는 사실만' 요약 규칙 (③의 핵심)
- 일어난 사건·수치·발언 주체·시점만 기술한다.
- 평가어("논란", "충격", "강력히")·추측·전망·기자 해석을 제거한다.
- 발언/주장은 **누가 말했는지 귀속**해서 적는다(사실과 주장 구분).
- 확인되지 않은 내용은 넣지 않는다. 한 항목 = 한두 문장.

### 2.5 나중에 고를 결정 포인트 (기획 확정 후 선택)
| 단계 | 결정할 것 | 후보 |
|---|---|---|
| ① 수집 | 고정 소스 목록 | 국내 언론 RSS(연합/한겨레/조선/한경 등), 해외(로이터/AP), 분야별 매체 |
| ② 선별 | 분야 구성 / 분야별 개수 / 선별 기준 | 정치·경제·사회·세계·IT 등 / 분야별 2~3건 / 최신·중복병합 |
| ② 선별 | **소스 신뢰도 순위** (중복 사건 시 어느 소스를 쓸지) | 재혁 × Claude가 함께 1·2·3순위 매겨 고정 |
| ④ 전달 | 제외 분야 설정 | 사용자가 특정 분야 끄기/켜기 (설정 한 곳에서) |
| ③ 요약 | 사용할 LLM | Claude API (모델은 빌드 시 최신 기준 선택) |
| ④ 전달 | 출력 위치·형식·알림 | `News/` 마크다운 (기본) / 추후 이메일·노션 |
| 자동화 | 아침 트리거 | macOS cron(launchd) / Claude Code `/schedule` |

### 2.6 성공 기준 (이게 되면 Phase 1 완료)
1. 명령 한 번(또는 자동 실행)으로 **그날짜 다이제스트 마크다운 파일이 생성**된다.
2. 내용이 **분야별로 정리**되고 각 항목에 **출처 링크**가 붙는다.
3. 요약이 **2.4 규칙(판단 없는 사실)**을 지킨다 — 표본 점검으로 확인.
4. 소스는 **고정 목록**에서만 오고, 날짜가 **실제 그날 전후**의 것이다(오래된 기사 혼입 없음).
5. **매일 아침 자동 실행**이 걸려 있다.

### 2.7 위험과 대응
- **요약이 사실을 왜곡/환각** → 원문 링크 필수 + 표본 점검 루틴 + 규칙 프롬프트 강화.
- **소스 RSS 구조 변경/중단** → 소스를 설정 한 곳에 모아 교체 쉽게, 실패한 소스는 건너뛰고 로그.
- **"사실만"의 경계 모호(선별=편집)** → 선별 기준을 ②에 명문화해 투명하게.
- **자동 실행 환경(컴퓨터 꺼짐 등)** → 자동화 수단 선택 시 함께 결정(클라우드 vs 로컬).

### 2.8 예외 처리 (엣지 케이스 정리)

> 원칙: **하나가 실패해도 전체가 멈추지 않는다.** 실패한 부분은 건너뛰고, 무슨 일이 있었는지 사용자가 알 수 있게 *명확한 문구*로 남긴다. 침묵하지 않는다.

#### A. 수집(①) 단계 변수
| 상황 | 원인 | 시스템 동작 | 사용자에게 보여줄 문구 |
|---|---|---|---|
| 소스 링크 접속 불가 | RSS 주소 변경·서버 다운·404·타임아웃 | 해당 소스만 건너뛰고 나머지로 계속 진행, 로그 기록 | ⚠️ `[연합뉴스] 소스에 접속할 수 없습니다. 오늘은 이 소스를 건너뜁니다.` |
| 모든 소스 실패 | 인터넷 끊김·전체 장애 | 다이제스트 생성 중단, 이전 파일은 그대로 보존 | ❌ `모든 뉴스 소스에 접속하지 못했습니다. 네트워크 연결을 확인해주세요. (기존 다이제스트는 유지됩니다)` |
| 소스는 살아있으나 새 기사 0건 | 발행 없음·날짜 필터에 걸림 | 그 분야는 "새 소식 없음"으로 표시 | 📭 `[경제] 오늘은 새로 수집된 기사가 없습니다.` |
| 분야별 기사가 목표 개수 미만 | 해당 분야 기사가 적음 | 있는 만큼만 싣기(빈자리 억지로 안 채움) | (별도 문구 없이 있는 만큼 표시) |
| 같은 사건이 여러 소스에 중복 | 동일 이슈를 여러 매체가 보도 | **소스 신뢰도 순위**에서 1순위 소스의 내용만 정리(나머지는 버림) | 🔗 `같은 사건이 여러 곳에서 보도됨 — [○○일보] 기준으로 정리했습니다.` |

#### B. 요약(③) 단계 변수
| 상황 | 원인 | 시스템 동작 | 사용자에게 보여줄 문구 |
|---|---|---|---|
| LLM API 호출 실패 | 네트워크·일시 오류 | 잠시 후 재시도(최대 N회), 그래도 실패 시 C 동작 | ⚠️ `요약 생성에 실패했습니다. 원문 제목과 링크만 표시합니다.` |
| API 키 미설정 | 첫 실행·키 누락 | 실행 중단 + 설정 방법 안내 | 🔑 `API 키가 설정되지 않았습니다. 설정 파일에 키를 입력한 뒤 다시 실행해주세요. (안내: …)` |
| API 사용 한도 초과 | 쿼터·결제 한도 | 실행 중단, 원문 링크만이라도 출력 | ⚠️ `요약 사용 한도를 초과했습니다. 오늘은 원문 링크만 제공합니다.` |
| 요약 결과가 비어있거나 형식 깨짐 | 모델 응답 이상 | 해당 항목은 원문 제목·링크로 대체 | ⚠️ `이 기사는 요약을 만들지 못해 원문으로 대체합니다.` |

#### C. 전달(④)·실행 단계 변수
| 상황 | 원인 | 시스템 동작 | 사용자에게 보여줄 문구 |
|---|---|---|---|
| 오늘 날짜 파일이 이미 존재 | 하루에 두 번 실행 | 기본은 덮어쓰기(최신 유지). 실행 전 안내 | ♻️ `2026-06-28 다이제스트가 이미 있습니다. 최신 내용으로 덮어씁니다.` |
| 저장 폴더 없음/권한 없음 | 경로 오류·권한 문제 | 폴더 생성 시도, 실패 시 중단 + 안내 | ❌ `News 폴더에 저장할 수 없습니다. 경로와 권한을 확인해주세요.` |
| 자동 실행 시각에 컴퓨터 꺼짐/절전 | 기기 상태 | (자동화 수단 선택 시 정책 결정) 다음 실행 때 당일분 보충 시도 | ℹ️ `어제(○○일) 다이제스트가 생성되지 않았습니다.` |
| 사용자가 과거 날짜로 수동 실행 | 날짜 인자 지정 | 그 날짜 기준으로 수집·생성(가능 범위 내) | ℹ️ `2026-06-20 기준으로 다이제스트를 생성합니다.` |
| 사용자가 특정 분야 제외 요청 | "○○ 분야 빼줘" 설정 | 설정에서 그 분야만 제외, 나머지 분야만 수집·정리 | ⚙️ `[스포츠] 분야를 제외하고 다이제스트를 생성합니다.` |

> 위 문구·재시도 횟수·"덮어쓰기 vs 새 버전" 같은 세부 정책은 구현 단계에서 최종 확정한다. 이 표는 **"어떤 변수를 반드시 다뤄야 하는가"의 체크리스트** 역할을 한다.

## 3. 개발 일정 (5일 · 이번 주 캘린더)

> 이번 주 안에 끝낸다. **월~목 작업 → 금 테스트.** 오늘(일)은 기획서 초안.

|---|---|---|---|---|---|
| **1** | 월 6/29 | 기획 확정 + 결정 | 기획서 최종 승인 + **결정 포인트 전부 확정**(소스 목록·분야 구성·소스 신뢰도 순위·LLM 모델·저장 위치·자동화 수단) | 2.5 표 빈칸이 모두 채워짐 | ⬜ |
