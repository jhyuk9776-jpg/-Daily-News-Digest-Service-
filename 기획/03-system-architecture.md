# 시스템 아키텍처

## 1. 전체 구조

```text
뉴스 출처
  ↓
수집(fetch)
  ↓
선별(select)
  ↓
요약(summarize)
  ↓
전달(deliver)
  ↓
News/YYYY-MM-DD.md
```

Phase 1에서는 복잡한 서버를 만들지 않고, 로컬 파이썬 스크립트와 스케줄러를 중심으로 시작한다.

## 2. 파이프라인

| 단계 | 역할 | 입력 | 출력 |
|---|---|---|---|
| 수집 | RSS/API에서 기사 후보를 가져온다. | 소스 설정 | 기사 리스트 |
| 선별 | 날짜, 분야, 중복, 우선순위를 기준으로 추린다. | 기사 리스트 | 선별 기사 리스트 |
| 요약 | AI가 사실 중심 요약을 생성한다. | 선별 기사 | 요약 기사 |
| 전달 | 날짜별 마크다운 파일을 만든다. | 요약 기사 | 다이제스트 파일 |

## 3. 기술 스택 초안

| 영역 | 1차 선택 | 비고 |
|---|---|---|
| 언어 | Python | 수집, 요약, 파일 생성에 적합 |
| 뉴스 수집 | RSS + requests/feedparser | API 비용 없이 시작 가능 |
| 본문 추출 | newspaper3k 또는 BeautifulSoup | 구현 단계에서 테스트 후 선택 |
| AI 요약 | OpenAI API 또는 Claude API | 비용, 품질, 사용 편의성 비교 필요 |
| 저장 | Markdown 파일 | Phase 1 기본 결과물 |
| 자동화 | macOS launchd 또는 GitHub Actions | 로컬 우선 후 클라우드 확장 가능 |
| DB | Supabase | 웹/앱 확장 시 도입 |

## 4. 요약 규칙

- 사실, 수치, 시점, 발언 주체를 우선한다.
- 평가어와 감정적 표현을 줄인다.
- 전망과 추측은 원문에 명시된 경우에만 `누가 말했다` 형태로 적는다.
- 원문에 없는 배경 지식을 덧붙이지 않는다.
- 불확실한 내용은 요약하지 않거나 출처 표현을 붙인다.

## 5. 예외 처리 원칙

- 하나의 소스가 실패해도 전체 실행을 멈추지 않는다.
- 모든 소스가 실패하면 새 파일 생성을 중단하고 명확한 오류를 출력한다.
- AI 요약이 실패한 기사는 제목과 원문 링크만 남긴다.
- 같은 날짜 파일이 이미 있으면 기본 정책을 정해야 한다. 후보는 덮어쓰기, 백업 후 생성, 건너뛰기다.

## 6. 나중에 확장할 구조

```text
Phase 1: Markdown 파일
Phase 2: Supabase 저장
Phase 3: 웹/PWA 화면
Phase 4: 이메일, 카카오톡, 푸시 알림
Phase 5: 불편/만족 신호 분석 및 아이디어 발굴
```
