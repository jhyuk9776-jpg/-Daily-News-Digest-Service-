## 0. 시작하기 전에 - 이야기의 3가지 핵심 질문

### i) 주인공이 원하는 게 무엇인가?

주인공은 매일 아침 짧은 시간 안에 세상에서 실제로 일어난 일을 판단과 해석이 덜어진 사실 중심 요약으로 파악하고 싶다.

더 깊은 욕구는 단순히 뉴스를 많이 읽는 것이 아니다. 세상의 흐름을 빠르게 잡고, 그 위에서 기획, 취업, 사업 아이디어를 발견하는 입력 루틴을 만드는 것이다.

### ii) 무엇이 그것을 방해하는가?

방해 요소는 사람이 아니라 세 가지 힘이다.

- 정보 과잉: 하루에도 너무 많은 뉴스가 쏟아진다.
- 편향과 잡음: 의견, 자극적 제목, 매체별 프레임이 사실 파악을 흐린다.
- 시간 부족: 여러 매체를 직접 확인하고 교차검증할 여유가 없다.

> 한마디로 적은 **편향된 정보 홍수**다.

### iii) 얻으면/못 얻으면 삶이 어떻게 달라지는가?

- 얻으면: 매일 5분으로 세상의 사실 골격을 잡고, 정보에 휩쓸리는 대신 흐름 위에서 문제와 기회를 보는 사람이 된다.
- 못 얻으면: 알고리즘 피드와 자극적 기사에 휩쓸리거나, 아예 세상의 흐름에서 멀어진다.

## 1. SB7 - 일곱 단계 서사

### 1.1 캐릭터

주인공은 세상의 흐름을 읽고 그 위에서 생각하고 싶은 사람이다. 현재 기준으로는 재혁처럼 취업, 기획, 문제 발굴, 사업 아이디어에 관심이 있지만 매일 뉴스를 일일이 챙길 시간은 부족한 사용자를 기준으로 한다.

> 주인공은 "뉴스를 많이 읽고 싶은 사람"이 아니라 **세상을 효율적으로 파악해 더 잘 생각하고 싶은 사람**이다.

### 1.2 난관에 직면한다

주인공은 매일 편향된 정보 홍수와 마주친다.

| 층위 | 문제 |
|---|---|
| 외적 문제 | 뉴스가 너무 많고 흩어져 있어 오늘 무슨 일이 있었는지 한눈에 보기 어렵다. |
| 내적 문제 | "나만 흐름에서 뒤처지는 것 아닌가"라는 불안과 피로가 생긴다. |
| 철학적 문제 | 사람은 누군가의 판단이 덧칠되기 전, 일어난 사실을 먼저 볼 수 있어야 한다. |

> 문제는 뉴스가 부족한 것이 아니라, 사실의 뼈대를 빠르게 잡기 어렵다는 점이다.

### 1.3 가이드를 만난다

데일리 사실 뉴스 다이제스트는 주인공을 대신해 판단하는 영웅이 아니라, 사실 파악을 돕는 가이드다.

- 공감: 매일 뉴스를 다 볼 시간은 없고, 의견보다 사실을 먼저 보고 싶다는 피로를 이해한다.
- 신뢰: 고정 뉴스 출처, 출처 링크, 날짜 필터, 중복 제거, 사실 중심 요약 규칙으로 일관성을 만든다.

> 시스템은 결론을 대신 내려주지 않고, 판단할 재료를 정리해주는 가이드다.

### 1.4 계획을 제시한다

가이드는 단순한 4단계 계획을 제시한다.

```text
수집 -> 선별 -> 요약 -> 전달
```

사용자가 할 일은 아침에 생성된 다이제스트를 5분간 읽는 것이다. 나머지는 시스템이 처리한다.

> 복잡한 뉴스 소비를 매일 반복 가능한 5분 루틴으로 바꾼다.

### 1.5 행동을 촉구한다

- 직접적 촉구: Phase 1에서 로컬 Python 스크립트로 매일 생성되는 마크다운 다이제스트를 만든다.
- 전환적 촉구: 맛보기 수준의 뉴스 요약에서 벗어나, 고정된 출처와 규칙으로 반복 가능한 시스템을 만든다.

> 다음 행동은 큰 플랫폼이 아니라, 매일 실행되는 작은 다이제스트 생성기를 완성하는 것이다.

### 1.6 실패를 피하게 도와준다

이 시스템이 없으면 사용자는 다음 실패를 반복할 수 있다.

- 정보가 너무 많아 오늘 중요한 사실조차 파악하지 못한다.
- 알고리즘 피드의 자극과 편향을 그대로 흡수한다.
- "언젠가 봐야지"를 반복하며 세상의 흐름에서 멀어진다.
- 기획과 아이디어 발굴의 입력 재료가 약해진다.

> 시스템은 매일 아침 사실의 골격을 제공해 흐름에서 밀려나는 실패를 줄인다.

### 1.7 성공으로 끝맺는다

성공한 사용자는 매일 아침 5분 안에 분야별 주요 사실을 확인한다. 필요하면 원문 링크로 깊게 들어가고, 평소에는 짧은 다이제스트만으로 세상의 방향을 놓치지 않는다.

이 토대가 안정되면 Phase 2에서 관심 분야, 이슈 그룹화, 불편/만족 신호 추출, 웹/PWA로 확장한다.

> 성공의 그림은 **많이 읽는 사람**이 아니라 **적게 읽고 더 잘 아는 사람**이다.

## 2. 제품 정의

### 2.1 프로젝트 이름

가칭: 데일리 사실 뉴스 다이제스트

### 2.2 한 줄 정의

매일 아침 신뢰할 수 있는 뉴스 출처에서 주요 사건을 수집하고, 중복을 줄인 뒤, AI가 판단과 해석을 덜어낸 사실 중심 요약으로 정리하는 뉴스 다이제스트 시스템.

### 2.3 핵심 가치

| 가치 | 설명 |
|---|---|
| 빠름 | 5분 안에 분야별 주요 사실을 확인한다. |
| 사실 중심 | 평가, 추측, 논평보다 사건, 발언 주체, 수치, 날짜를 우선한다. |
| 반복 가능 | 매일 같은 기준과 형식으로 생성된다. |
| 출처 확인 | 모든 요약에 원문 링크를 남긴다. |
| 확장 가능 | 마크다운에서 시작해 웹, 앱, 알림으로 확장한다. |

### 2.4 제품 원칙

1. 사용자는 뉴스를 많이 읽는 사람이 아니라, 세상을 효율적으로 파악하고 싶은 사람이다.
2. 요약은 판단을 대신 내려주지 않는다. 판단할 재료를 정리한다.
3. 출처 링크는 반드시 남긴다.
4. 요약 품질보다 먼저 중요한 것은 출처, 날짜, 중복 제거의 신뢰성이다.
5. 작은 결과물을 매일 쓸 수 있게 만든 뒤 확장한다.

## 3. 기술 설계

### 3.1 범위 / 비범위

| 구분 | 내용 |
|---|---|
| 이번에 만든다 | 분야별 사실 중심 데일리 다이제스트, RSS/API 수집, 선별, 중복 제거, AI 요약, 마크다운 저장 |
| 이번엔 안 만든다 | 웹/앱 UI, 개인화 추천, 푸시 알림, 의견/해설 생성, 커뮤니티 반응 분석, 유료 뉴스 전문 스크래핑 |
| 설계 원칙 | 수집, 선별, 요약, 전달을 독립 부품으로 나눠 교체 가능하게 만든다. |

### 3.2 접근 방식

Phase 1은 로컬 Python 스크립트로 시작한다.

```text
fetch -> select -> summarize -> deliver
```

선정 이유:

- 구현이 빠르고 통제하기 쉽다.
- API 비용 외 운영 비용이 거의 없다.
- 스크립트 단위로 테스트하기 좋다.
- 나중에 Supabase, 웹/PWA, 앱으로 확장할 때도 데이터 파이프라인을 재사용할 수 있다.

### 3.3 파이프라인

| 단계 | 하는 일 | 입력 -> 출력 | 부품 경계 |
|---|---|---|---|
| 수집(fetch) | 고정 소스에서 어제~오늘 기사 수집 | 소스 목록 -> 기사 리스트 | 소스 추가/교체는 이 단계에서만 처리 |
| 선별(select) | 날짜, 분야, 중복, 우선순위 기준으로 기사 정리 | 기사 리스트 -> 선별 기사 리스트 | 선별 기준은 이 단계에 모음 |
| 요약(summarize) | AI가 사실 중심 규칙으로 압축 | 선별 기사 -> 요약 기사 | 모델과 프롬프트를 교체 가능하게 유지 |
| 전달(deliver) | 날짜별 마크다운 저장 | 요약 기사 -> `News/YYYY-MM-DD.md` | 전달 채널 추가는 이 단계에서 처리 |

> 각 부품은 독립적으로 테스트 가능해야 한다.

### 3.4 요약 규칙

- 일어난 사건, 수치, 발언 주체, 시점을 우선한다.
- 평가어, 감정적 표현, 기자 해석을 줄인다.
- 전망과 추측은 원문에 명시된 경우에만 `누가 말했다` 형태로 적는다.
- 원문에 없는 배경 지식을 덧붙이지 않는다.
- 확인되지 않은 내용은 넣지 않는다.
- 한 항목은 1~3문장으로 유지한다.

피해야 할 표현:

- 논란이 커지고 있다
- 충격을 주고 있다
- 큰 파장이 예상된다
- 업계가 주목하고 있다

### 3.5 결과물 형식 초안

```markdown
# 데일리 뉴스 다이제스트 - 2026-06-29

## 경제

### 기사 제목
- 요약 1
- 요약 2
- 요약 3
- 출처: [매체명](원문 링크)

## IT/테크

오늘 수집된 주요 기사가 없습니다.
```

### 3.6 RSS 수집 전략

RSS는 뉴스 사이트의 최신 기사 목록을 기계가 읽기 쉬운 형식으로 제공하는 피드다. 보통 제목, 링크, 발행 시간, 요약문이 들어 있고, 원문 전문은 언론사 정책에 따라 제공되지 않을 수 있다.

Phase 1에서는 RSS를 뉴스 수집의 기본 입구로 사용한다. 직접 크롤링보다 구현이 쉽고, 원문 링크 중심으로 처리할 수 있어 저작권과 운영 부담이 낮다.

#### RSS 사용 규모 참고

| 질문 | 확인 결과 |
|---|---|
| RSS는 아직 쓰이나? | 그렇다. BuiltWith 기준 RSS를 사용하는 라이브 웹사이트가 38,136,800개, 한국 라이브 사이트가 83,842개로 집계되어 있다. |
| 보통 몇 개의 RSS를 쓰나? | 공개된 표준값은 찾지 못했다. 개인 RSS 리더는 사용자가 원하는 만큼 구독하고, 뉴스 수집 시스템은 목적에 따라 소스 수가 달라진다. |
| 어느 RSS가 가장 많이 쓰이나? | 언론사별 RSS 구독 순위를 신뢰성 있게 공개한 최신 데이터는 찾지 못했다. |
| 우리 MVP는 몇 개로 시작하나? | 처음에는 분야별 2~3개, 전체 8~12개 RSS로 시작한다. 품질 확인 후 늘린다. |

참고:

- BuiltWith RSS Usage Statistics: https://trends.builtwith.com/feeds/RSS
- SBS 뉴스 RSS 안내: https://news.sbs.co.kr/news/rss.do
- RSS 개념 참고: https://en.wikipedia.org/wiki/RSS

#### Phase 1 RSS 운영 원칙

1. 분야별로 최소 2개 이상의 출처를 둔다.
2. 처음부터 많이 넣지 않고 전체 8~12개 RSS로 시작한다.
3. 같은 사건이 여러 출처에 나오면 중복 제거 후 우선순위가 높은 출처를 대표 링크로 쓴다.
4. RSS 이용 조건이 명시된 곳은 개인/비상업 사용 범위에서만 쓴다.
5. 상업 서비스로 확장할 때는 언론사 제휴나 API 사용 조건을 다시 확인한다.
6. 본문 전문 저장보다 제목, 요약, 발행 시간, 링크 중심으로 시작한다.

#### 1차 RSS 후보군

아래는 확정 목록이 아니라 검증할 후보군이다. 실제 구현 전 각 RSS URL이 정상 작동하는지 확인한다.

| 분야 | 후보 방향 | 예시 |
|---|---|---|
| 경제 | 국내 경제 RSS + 해외 경제/비즈니스 RSS | SBS 경제, 국내 경제지, Reuters/BBC/Guardian 경제 후보 |
| 사회 | 국내 방송/종합지 사회 RSS | SBS 사회, KBS/MBC/JTBC/한겨레/경향 후보 |
| 세계 | 국내 국제 RSS + 해외 월드 뉴스 RSS | SBS 국제, BBC World, Reuters World 후보 |
| IT/테크 | 국내 IT 매체 + 해외 테크 매체 RSS | 국내 IT 매체, The Verge/TechCrunch 후보 |

#### 시작 추천안

처음에는 다음처럼 작게 시작한다.

| 구분 | 개수 |
|---|---:|
| 경제 | 2~3개 RSS |
| 사회 | 2~3개 RSS |
| 세계 | 2~3개 RSS |
| IT/테크 | 2~3개 RSS |
| 합계 | 8~12개 RSS |

이 정도면 하루 기사 후보가 충분히 모이면서도 중복 제거와 요약 품질을 눈으로 확인하기 쉽다.

## 5. 성공 기준

Phase 1은 아래 조건을 만족하면 완료로 본다.

1. 명령 한 번 또는 자동 실행 한 번으로 오늘 날짜 다이제스트 마크다운 파일이 생성된다.
2. 실제 날짜 기준으로 오늘 또는 전날 주요 기사만 포함된다.
3. 내용이 분야별로 정리된다.
4. 모든 항목에 원문 링크가 있다.
5. 요약이 사실 중심 규칙을 지킨다.
6. 일부 소스 실패, 요약 실패, 새 기사 없음이 문서에 명확히 표시된다.
7. 결과물을 재혁이 매일 5분 안에 읽을 수 있다.

## 6. 위험과 대응

| 위험 | 대응 |
|---|---|
| 요약이 사실을 왜곡하거나 환각한다 | 원문 링크 필수, 표본 점검, 프롬프트 강화 |
| RSS 구조가 바뀌거나 중단된다 | 소스를 설정 파일에 모으고 실패한 소스는 건너뛰기 |
| 오래된 기사가 섞인다 | 날짜 필터를 강하게 적용 |
| 같은 사건이 중복된다 | URL, 제목 유사도, 출처 우선순위 기준으로 병합 |
| 사실과 해석의 경계가 흐려진다 | 요약 규칙과 선별 기준을 문서화 |
| 자동 실행 시 컴퓨터가 꺼져 있다 | GitHub Actions 또는 보충 실행 정책 검토 |
| 비용이 커진다 | 기사 수 제한, 캐시, dry-run, 저비용 모델 사용 |

## 7. 예외 처리

원칙: 하나가 실패해도 전체가 멈추지 않는다. 실패한 부분은 건너뛰고, 무슨 일이 있었는지 명확한 문구로 남긴다.

### 7.1 수집 단계

| 상황 | 시스템 동작 | 사용자에게 보여줄 문구 |
|---|---|---|
| 소스 접속 불가 | 해당 소스만 건너뛰고 계속 진행 | `[연합뉴스] 소스에 접속할 수 없습니다. 오늘은 이 소스를 건너뜁니다.` |
| 모든 소스 실패 | 새 파일 생성을 중단하고 기존 파일 보존 | `모든 뉴스 소스에 접속하지 못했습니다. 네트워크 연결을 확인해주세요.` |
| 새 기사 0건 | 해당 분야를 빈 상태로 표시 | `[경제] 오늘은 새로 수집된 기사가 없습니다.` |
| 분야별 기사 부족 | 있는 만큼만 표시 | 별도 문구 없이 있는 만큼 표시 |
| 중복 기사 발견 | 우선순위 높은 출처 기준으로 정리 | `같은 사건이 여러 곳에서 보도되어 [매체명] 기준으로 정리했습니다.` |

### 7.2 요약 단계

| 상황 | 시스템 동작 | 사용자에게 보여줄 문구 |
|---|---|---|
| LLM API 호출 실패 | 재시도 후 실패하면 원문 제목과 링크만 표시 | `요약 생성에 실패했습니다. 원문 제목과 링크만 표시합니다.` |
| API 키 미설정 | 실행 중단 후 설정 안내 | `API 키가 설정되지 않았습니다. 설정 파일을 확인해주세요.` |
| API 사용 한도 초과 | 요약 없이 원문 링크 중심 출력 | `요약 사용 한도를 초과했습니다. 오늘은 원문 링크만 제공합니다.` |
| 요약 형식 깨짐 | 해당 항목을 원문 제목과 링크로 대체 | `이 기사는 요약을 만들지 못해 원문으로 대체합니다.` |

### 7.3 전달/실행 단계

| 상황 | 시스템 동작 | 사용자에게 보여줄 문구 |
|---|---|---|
| 오늘 날짜 파일이 이미 존재 | 기본은 최신 내용으로 덮어쓰기 | `오늘 다이제스트가 이미 있습니다. 최신 내용으로 덮어씁니다.` |
| 저장 폴더 없음 | 폴더 생성 시도, 실패 시 중단 | `News 폴더에 저장할 수 없습니다. 경로와 권한을 확인해주세요.` |
| 자동 실행 누락 | 다음 실행 때 보충 생성 검토 | `어제 다이제스트가 생성되지 않았습니다.` |
| 특정 분야 제외 요청 | 설정에서 해당 분야 제외 | `[스포츠] 분야를 제외하고 다이제스트를 생성합니다.` |
