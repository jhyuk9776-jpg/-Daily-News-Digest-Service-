## 3.1 RSS 출처 결정 기준

RSS 출처는 아직 확정하지 않았다. 먼저 RSS가 안정적으로 제공되는지, 이용 조건이 개인/비상업 사용을 허용하는지, 분야별 중복을 얼마나 줄일 수 있는지 확인한다.

| 항목 | 결정 기준 |
|---|---|
| 시작 개수 | 전체 8~12개 RSS |
| 분야별 개수 | 경제, 사회, 세계, IT/테크별 2~3개 |
| 우선순위 | 공신력, 원문성, RSS 안정성, 발행 시간 정확도 |
| 사용 방식 | 제목, 요약, 발행 시간, 원문 링크 중심 |
| 제외 기준 | RSS 불안정, 이용 조건 불명확, 광고/연예/스포츠 중심, 중복 과다 |
| 상업화 시 | 언론사 제휴/API/이용약관 재검토 |

조사 메모:

- BuiltWith 기준 RSS 사용 라이브 웹사이트는 38,136,800개, 한국 라이브 사이트는 83,842개로 집계되어 있다.
- SBS는 RSS 서비스를 제공하며, 개인 이용자의 비상업적 사용만 허가한다고 안내한다.
- 언론사별 RSS 구독 순위나 "가장 많이 쓰이는 RSS"에 대한 신뢰 가능한 공개 최신 순위는 찾지 못했다.
- 따라서 Phase 1은 인기 순위보다 공신력, 안정성, 분야 적합성을 기준으로 RSS를 고른다.

## 3.2 웹/앱 언어·스택 결정 (2026-06-29 확정)

웹과 앱을 함께 운영하기 위한 언어·스택을 **TypeScript(React 계열)**로 확정한다.

| 항목 | 결정 |
|---|---|
| 언어 | TypeScript |
| 웹 | Next.js (PWA 포함) |
| 앱 | Expo / React Native |
| 백엔드 | 기존 Python 파이프라인 유지 |
| DB 연동 | Supabase (Phase 2 도입) |

결정 이유:

- 언어 하나로 웹+iOS+안드로이드를 모두 커버해 솔로 개발 부담이 가장 작다.
- Supabase 공식 지원(JS SDK)이 가장 두터워 Phase 2 DB 결정과 직결된다.
- 생태계·자료·AI 코딩 도구 지원이 넓어 혼자서도 빠르게 진행 가능하다.
- 뉴스 다이제스트는 텍스트·리스트 중심 읽기 앱이라 웹 SEO/공유에 강한 스택이 유리하다.

실행 방식:

1. **Next.js로 PWA 웹을 먼저 출시**해 사용성을 검증한다.
2. 검증 후 같은 TypeScript 자산으로 **Expo 네이티브 앱**을 확장한다.
3. 이 작업은 **Phase 3 이후**다. 지금은 Python 데이터 파이프라인 완성이 우선이며, 이 결정은 기록만 해둔다.
