# 기획서 — 데일리 뉴스 다이제스트 (상업화 / 서비스 기획)

작성일: 2026-07-02
작성: 이재혁

> 이 문서는 **서비스·상업 기획**이다. 왜·누구를 위해·어떻게 판매하는가를 다룬다.
> 기술 구현의 부품·일정은 `기획/구현기획/`과 `docs/superpowers/`를 참조한다.
> 가격·인원 등 숫자는 대부분 `미정`이며, `수익·원가 기획`에서 확정한다.

## 0. 시작하기 전에 — 이야기의 3가지 핵심 질문

**i) 주인공이 원하는 게 무엇인가?**
- 표면 욕구: 아침에 **짧은 시간 안에** 세상 흐름과 자기 분야 소식을 파악한다.
- 더 깊은 욕구: 정보 홍수에 뒤처지지 않는다는 **안심**. "중요한 걸 놓치지 않았다"는 확신.

**ii) 무엇이 그것을 방해하는가?** (적 = 2~3개의 힘)
- **정보 과잉** — 볼 건 너무 많고 시간은 없다.
- **판단의 오염** — 대부분의 요약·뉴스가 평가·전망·낚시로 사실을 흐린다.
- **신뢰의 부재** — 출처가 불명확하고, 무엇이 교차검증된 사실인지 알 수 없다.

> 적을 한마디로: **"믿을 수 없는 정보 홍수."**

**iii) 얻으면/못 얻으면 삶이 어떻게 달라지는가?**
- 성공: 매일 아침 5분, **판단이 걷힌 사실만** 출처와 함께 확인하고 하루를 시작한다.
- 실패: 낚시성 요약에 시간을 쓰고도 "제대로 안 것 같지 않은" 불안이 남는다.

## 1. SB7 — 일곱 단계 서사

### ① 캐릭터(주인공)
**30~40대 지식노동자·직장인.** 뉴스를 "많이 읽고 싶은" 사람이 아니라, **자기 일·관심
분야에서 흐름을 놓치면 안 되는데 그럴 시간이 없는** 사람. (뉴스 친화도가 가장 높은 연령대이며,
이들이 많이 쓴다고 알려지면 다른 연령대가 자연히 따라온다는 가정.)
- 1차 타깃: 위 A 세그먼트.
- 확장 타깃: B(투자·리서치·의사결정자), C(업계 종사자·B2B 팀). 신뢰성 차별점이 강하게
  먹히지만 정확도 바·영업 사이클이 커서 **후순위**.

> 주인공은 "효율적으로 세상 흐름을 파악하고 싶은 바쁜 30~40대"다.

### ② 난관에 직면
- **외적 문제:** 매일 쏟아지는 기사를 다 볼 수 없다. 무료 요약은 분야별 2건뿐이라 깊이가 얕다.
- **내적 문제:** "중요한 걸 놓쳤을지 모른다"는 불안. 요약을 봐도 믿어도 되는지 확신이 없다.
- **철학적 문제:** 정보 서비스는 **판단을 대신 내려주면 안 된다.** 판단할 **재료**를 정직하게
  줘야 한다. 낚시·전망·평가로 사실을 흐리는 것은 사용자의 판단력을 빼앗는 일이다.

### ③ 가이드를 만난다
시스템은 영웅이 아니라 **가이드**다. 신뢰를 이렇게 준다:
- **공감:** "시간은 없는데 놓치긴 싫다"는 고통을 안다. 그래서 아침 정시에 짧게, 골라서 준다.
- **권위/신뢰(명문화된 규칙):**
  - 판단 없는 **사실 중심 요약**(평가어·전망 배제, AI_CONTEXT §6).
  - **출처 1:1 링크** — "출처 없는 사실은 싣지 않는다."
  - **교차검증**(여러 독립 매체가 보도한 사건 우선) + **객관성 점수**(평가어 감점, observe-only 누적).
  - **실패를 숨기지 않음** — 사실을 못 뽑은 기사는 억지로 싣지 않고 제외하며, 실패를 기록·통지한다.

> 우리는 답을 주는 영웅이 아니라, **믿을 수 있는 재료를 정직하게 골라주는 가이드**다.

### ④ 계획을 제시 (주인공이 따를 단순한 계획)
1. **무료로 시작** — 매일 아침 분야별 2건(총 8건) 사실 요약을 받는다.
2. **내 방식대로** — 총 8건 안에서 원하는 분야에 더 배분한다(커스텀, 원가 0 → 무료 제공).
3. **더 깊게(유료)** — 정액제로 **총 상한 N**을 늘려 관심 분야를 더 많이 받는다.

요금제 (3티어, 총 N + 사용자 배분. 숫자 **미정** — 수익·원가 기획에서 확정):

| 티어 | 총 상한 N | 배분 | 성격 |
|---|---|---|---|
| 무료 | 8 (기본 분야별 2) | 8 안에서 커스텀 가능 | 습관 형성·맛보기 |
| Basic | 미정(~15) | 사용자 커스텀 | 내 분야 깊이 |
| Pro | 미정(~25) | 사용자 커스텀 | 여러 분야 모니터링 |
| Max | 미정(~40) | 사용자 커스텀 | 헤비 유저·확장 |

> 무료는 **커스텀(배분)**, 유료는 **볼륨(총 N)**. 두 가치 축을 분리한다.

### ⑤ 행동을 촉구
- **직접적 촉구:** "지금 무료로 오늘 아침 다이제스트를 받아보세요."
- **전환적 촉구:** 무료로 습관이 붙고 8건 한도에 부딪히면 → "더 깊게 보려면 업그레이드."

### ⑥ 실패를 피하게
안 쓰면 잃는 것: 낚시성 정보에 시간을 낭비하고도 남는 불안, 중요한 흐름을 놓칠 위험.
시스템이 막는 법: 판단을 걷어낸 사실 + 출처 + 교차검증으로 **"짧지만 믿을 수 있는 아침"**을 보장.

### ⑦ 성공으로 끝맺음
- 주인공: 매일 아침 5분에 자기 분야 흐름을 **사실 기준으로** 파악하고 안심하며 하루를 연다.
- 사업: 무료 유저의 한계비용이 0에 가까워, **소수의 유료 전환만으로 흑자**가 되는 구조.

> 성공 = 바쁜 30~40대가 "아침 뉴스는 여기"라고 습관화하고, 그중 일부가 깊이를 위해 결제한다.

## 2. 기술 설계 (상업 시스템 관점)

### 2.1 범위 / 비범위
- **범위(상업화):** 3티어 정액제, 총 N 상한 + 사용자 배분, 공유 풀+캐시 원가 구조, 결제·계정.
- **비범위(당장 아님):** B2B 팀 요금, 다국어, 실시간 푸시, 개인화 추천 알고리즘.
- **설계 원칙:** 부품화(수집·선별·요약·전달 분리 유지), 무료 한계비용 최소화, 실패 투명.

### 2.2 접근 방식 — 공유 풀 + 캐시 (핵심 원가 전략)
하루에 **넉넉한 공통 요약 풀**(예: 분야별 상위 N건, 미정)을 **한 번만** 생성해 캐시한다
(`cache/summaries.json` 재사용). 각 사용자의 무료/유료/커스텀 선택은 **캐시에서 고르기**다.
- 이유: AI 비용이 **사용자 수가 아니라 하루 풀 크기에만** 비례 → 무료 유저 한계비용 ≈ 0.
- 대안(기각): 사용자별 개별 생성 → 무료 유저 수에 비례해 적자. 구조적으로 위험.

### 2.3 파이프라인 / 아키텍처 (상업 확장 지점)
| 단계 | 하는 일 | 입력→출력 | 상업화 시 변화 |
|---|---|---|---|
| 수집·선별·요약 | 공통 풀 생성 | RSS → 캐시된 요약 풀 | 풀 크기를 상한에 맞게 확대 |
| 저장 | 결과 보존 | 파일 → **DB(Supabase)** | 다중 사용자·계정별 설정 저장 |
| 배분·전달 | 사용자별 뷰 구성 | 캐시 + 사용자 설정 → 개인 다이제스트 | 티어·커스텀 배분 적용 |
| 계정·결제 | 구독 관리 | 사용자 → 티어 상태 | 신규(PG 연동) |
| 스케줄 | 매일 정시 실행 | cron | Actions → VM/서버리스 |

### 2.4 핵심 규칙 (본질 방어)
- 판단 없는 사실만. 출처 1:1. 교차검증·객관성 우선. **못 뽑은 사실은 억지로 싣지 않는다.**
- 무료 유저에게도 커스텀(배분)을 준다(원가 0). 유료는 볼륨으로만 구분.

### 2.5 결정 포인트 (미정 → 이후 확정)
| 항목 | 결정할 것 | 후보/메모 |
|---|---|---|
| 요약 단가 | Replicate Haiku 실단가 | 확인 필요 (수익·원가 기획) |
| 요금 | 티어별 월 정액 | 미정 (₩ 단위, 원가+마진) |
| 총 상한 N | Basic/Pro/Max 값 | 미정 (~15/~25/~40 초안) |
| 공통 풀 크기 | 하루 생성 요약 수 | 미정 (커버율 vs 원가) |
| 결제 | PG 사업자 | 미정 (수수료 ~3.5% 가정) |
| 저작권 | 유료 시 매체 사용 범위 | **법무·저작권 조사 필요**(§2.7) |

### 2.6 성공 기준
1. 무료 다이제스트가 매일 정시에 안정 발송된다(현 파이프라인 기반).
2. 공유 풀+캐시로 **무료 유저 한계비용이 0에 수렴**함을 원가 모델로 검증.
3. 결제·계정·티어 상한이 동작한다.
4. **흑자 전환 인원**이 한 자릿수~십수 명 수준으로 계산됨(원가 기획에서 확정).
5. 유료 사용자가 커스텀 배분으로 자기 분야를 실제로 더 깊게 받는다.

### 2.7 위험과 대응
| 위험 | 대응 |
|---|---|
| **저작권/약관** — 유료 전환 시 "개인·비상업" 허가 RSS 위반(예: SBS는 비상업만 허가, 결정 로그 1.3) | `legal-copyright` 에이전트로 매체별 약관 조사 → 제휴/API 계약, 발췌 대신 사실 재구성+링크, 위반 매체 제외. **전문 변호사 검토 필요.** |
| **원가 폭증** — 사용자별 생성 시 무료 유저에 비례해 적자 | 공유 풀+캐시로 한계비용 0화(§2.2) |
| **낮은 전환율** — 무료만 쓰고 결제 안 함 | 한계비용 0라 무료가 늘어도 손실 아님. 전환은 "볼륨" 욕구로 유도 |
| **품질 사고가 유료 신뢰 훼손** | 실패 제외·로그·알림 + 요약 과반 실패 가드(구현 완료) |
| **개인정보/전자상거래 규제** — 결제·이메일 수집 | 개인정보처리방침·이용약관·환불정책 필요(법무 트랙) |

### 2.8 예외 처리
| 상황 | 원인 | 시스템 동작 | 사용자에게 |
|---|---|---|---|
| 사실 못 뽑음 | 본문 추출/모델 0불릿 | 다이제스트에서 제외 + 실패 로그 | (해당 기사 미표시) |
| 분야 기사 부족 | 그날 후보 < 상한 | 있는 만큼만(억지 채움 없음) | 📭 "오늘 이 분야 새 기사가 적습니다" |
| 요약 과반 실패 | 토큰·API 붕괴 | 비정상 종료, 부실본 배포 차단 | ⚠️ 발송 보류·관리자 통지 |
| 커스텀 배분 초과 | ∑ > 총 N | 저장 거부 | ℹ️ "총 N건을 넘을 수 없습니다" |

## 3. 개발 일정 (상업화 로드맵 — Phase 단위)

```
 Phase 1  ✅  사실 중심 무료 다이제스트 + 자동화 (완료)
 Phase 1.5 ⏸  분야별/총 상한 골격 (limits.yaml 완료, 총 N·프리셋 확장 예정)
 Phase 2  ⬜  Supabase 저장 · 계정 · 객관성/실패 데이터 활용
 Phase 3  ⬜  웹/PWA·앱 · 커스텀 배분 UI · 결제(PG) · 요금제 오픈
```

| Phase | 단계 | 하는 일 | 완료 조건 | 상태 |
|---|---|---|---|---|
| 1 | 파이프라인·자동화 | 수집→요약→매일 발송 | 매일 다이제스트 생성 | ✅ |
| 1.5 | 상한 골격 | limits.yaml, 총 N·프리셋 | 상한값으로 볼륨 조절 | ⏸️ |
| 2 | 데이터·계정 | Supabase, 계정, 점수/실패 축적 활용 | 다중 사용자 저장 | ⬜ |
| 3 | 제품·결제 | 웹/앱, 커스텀 배분 UI, PG 결제 | 유료 결제로 상한 해제 | ⬜ |

> 병목: **Phase 3의 결제·저작권**. 돈을 받는 순간 약관·규제가 실질 리스크가 된다 — 법무 트랙을 Phase 2와 병행해 미리 조사한다.
