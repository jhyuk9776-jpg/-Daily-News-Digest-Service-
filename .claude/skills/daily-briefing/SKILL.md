# Daily Briefing (작업 시작 브리핑)

## Overview

매일 작업을 시작하기 전, 흩어진 기획 문서·진행 상태·미정 결정을 한 화면으로 모아
브리핑한다. 목적은 사용자가 "오늘 어디서부터, 무엇을, 무엇을 정하고 시작하는지"를
30초 안에 파악하게 하는 것.

**원칙:** 날짜만 믿지 말고 실제 산출물로 진행 위치를 판단한다. 일정은 밀릴 수 있다.

## Procedure

1. **맥락 읽기** — 다음을 읽는다:
   - `AI_CONTEXT.md` (Phase 1 범위·원칙)
   - `기획/05-roadmap.md` (Day별 목표/완료 조건)
   - `기획/04-decision-log.md` (확정/미정 결정)
   - `기획/회의내용정리/` 최신 메모, `기획/시스템기획/기능설계/` 최신 설계
2. **진행 위치 판단** — `git log --oneline -10` 과 실제 파일(`src/`, `raw/`, 스펙)을
   대조해 "지금 며칠차의 어디"인지 정한다. 오늘 날짜(`date +%F`)는 참고만.
3. **브리핑 출력** — 아래 4개 섹션으로, 짧게.
4. **시작 확인** — 마지막에 "오늘 이 순서로 시작할까요?"와
   "먼저 정해야 할 결정"을 물어 사용자 승인을 받는다. 승인 전 구현 시작 금지.

## Output Format

```markdown
# 📅 오늘의 브리핑 — YYYY-MM-DD (Day N: <로드맵 단계명>)

## 1. 오늘 위치 + 목표
- 로드맵상 오늘 목표 1줄
- 완료 조건: ...

## 2. 지난 작업 요약
- 어제까지 완료한 것 (git/산출물 기준, 2~4줄)
- 남은/이월된 것

## 3. 오늘 할 작업
- [ ] 작업 1
- [ ] 작업 2
- (Day 범위를 넘는 작업은 "오늘 범위 아님"으로 명시)

## 4. 결정 필요 항목
| 항목 | 현재 상태 | 추천안 |
|---|---|---|
| ... | 미정 | ... |
(없으면 "오늘 막는 미정 결정 없음"이라고 적는다)
```

## Common Mistakes

- ❌ 날짜만 보고 Day를 단정 — 진행이 밀렸는데 다음 Day를 브리핑하면 어긋난다. 산출물로 확인.
- ❌ 결정 항목을 나열만 하고 추천을 안 함 — 항상 추천안을 같이 준다.
- ❌ 브리핑이 길어짐 — 각 섹션 핵심만. 세부는 물어보면 펼친다.
- ❌ 승인 없이 바로 구현 — 브리핑 후 사용자 확인을 받고 시작한다.
